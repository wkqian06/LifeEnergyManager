# LifeEnergyManager Scheduled Task Setup

Use this file when creating Codex scheduled tasks for a configured LifeEnergyManager workspace.

## Task 1: Morning Planning

Name:

`LifeEnergyManager morning planning`

Cadence:

- Monday-Saturday.
- User's configured morning planning time.
- User's configured timezone.

Prompt:

```text
Run LifeEnergyManager morning planning for today.

Read prompts/subagents.md, outputs/life_energy_tracker.md, and the current plan files under outputs/. Follow prompts/morning.md exactly:
- read the tracker and rolling state first,
- ask whether I have extra urgent, external, or tempting tasks today,
- call UrgencyTriageAgent if I provide extra urgent, external, or tempting tasks and subagent tools are available,
- call DailyPlannerAgent for provisional plan options when subagent tools are available,
- call AdviceAgent for status summary, today advice, and anti-distraction tip when subagent tools are available,
- record unavailable fallback and complete the same structured pass in the main thread if any required subagent tool is unavailable,
- triage extra tasks before accepting them and keep final acceptance in the main thread,
- produce a provisional plan with the required Subagent calls audit block,
- wait for my confirmation before generating HTML or PNG artifacts,
- after confirmation, generate the daily HTML workbench and wallpaper under outputs/,
- call ArtifactQAAgent for artifact QA when subagent tools are available,
- record unavailable fallback and complete artifact QA in the main thread if ArtifactQAAgent is unavailable,
- QA the artifacts before presenting them.
```

## Task 2: Evening Check-In

Name:

`LifeEnergyManager evening check-in`

Cadence:

- Monday-Saturday.
- User's configured evening check-in time.
- User's configured timezone.

Prompt:

```text
Run LifeEnergyManager evening check-in for today.

Read prompts/subagents.md first. Ask me for the Markdown report generated by today's HTML workbench under outputs/daily-workbenches/. Follow prompts/evening.md exactly:
- use the workbench report as the primary source,
- update daily log, rolling 30-day state, active micro-sprints, and temporary urgent tasks,
- save any standalone daily report under outputs/daily-reports/,
- call EnergyQuantAgent when enough report content exists and subagent tools are available,
- if only sparse data exists, ask for the minimal evening fields before energy inference,
- record unavailable fallback and infer beta next-day drive-resistance state in the main thread if EnergyQuantAgent is unavailable,
- never treat next-day drive-resistance state as diagnosis,
- define score direction as 0 = tomorrow's motivation/willingness is strong and 100 = tomorrow is likely to feel resistant, unwilling, or hard to start,
- save user drive-resistance self-score if present,
- generate tomorrow's first-action seed and likely focus mode,
- include the required Subagent calls audit block.
```

## Task 3: Sunday Review

Name:

`LifeEnergyManager Sunday review`

Cadence:

- Sunday.
- User's configured weekly review time.
- User's configured timezone.

Prompt:

```text
Run LifeEnergyManager Sunday light review.

Read prompts/subagents.md first. Follow prompts/sunday_review.md exactly:
- read outputs/life_energy_tracker.md and the last 7 daily logs from outputs/,
- call WeeklyReviewAgent before updating next week's plan when subagent tools are available,
- record unavailable fallback and complete the same structured weekly review in the main thread if WeeklyReviewAgent is unavailable,
- summarize the week,
- compress rolling 30-day state where appropriate,
- update next week's plan,
- list agent-delegable tasks,
- choose Monday's first action,
- keep the review light and do not create a full daily plan unless I ask,
- include the required Subagent calls audit block.
```

## Notes

- Prefer one persistent Codex thread for the workflow so the user can review the same planning history.
- Do not create duplicate automations if one already exists.
- Preserve the user's configured times and timezone when updating an existing automation.
- Required subagent calls are mandatory when the environment supports subagents; otherwise the scheduled task must log `unavailable fallback` and continue in the main thread.
